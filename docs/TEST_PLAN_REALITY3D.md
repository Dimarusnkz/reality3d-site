# План тестирования Reality3D.ru (минимум для запуска в прод)

Цель: проверить основные бизнес‑процессы сайта и готовность к полноценной работе.

## 1) Критические сценарии (P0)

### 1.1 Магазин: полный цикл заказа
1. Товар создан и виден на витрине (админ → магазин → товары).
2. Клиент (авторизованный) добавляет товар в корзину.
3. Клиент оформляет заказ на `/checkout`.
4. Заказ создаётся, товар резервируется.
5. Клиент переходит на страницу заказа `/shop/order/[id]`.
6. Клиент оплачивает по ссылке/QR (Т‑Банк перевод).
7. Администратор подтверждает оплату в админке.
8. Система:
   - Переводит заказ/оплату в “Оплачен”
   - Создаёт запись платежа
   - Делает списание со склада
   - Создаёт приход в кассу
9. Администратор ставит отгрузку (трек/перевозчик/статус доставки).
10. Клиент видит актуальные статусы (на русском, цветом).

### 1.2 Склад: приход → расход → инвентаризация
1. Создать приход (документ), добавить позицию, провести.
2. Проверить рост остатков и корректность `stock` и `reserved`.
3. Выполнить списание (в т.ч. по причине “брак”).
4. Провести инвентаризацию и убедиться, что остатки пересчитались.
5. Проверить логи склада.

### 1.3 Касса: операции и отражение оплаты заказа
1. Ручная операция (приход/расход) создаётся и отображается человеко‑читаемо.
2. Подтверждение оплаты заказа создаёт кассовую операцию `Оплата заказа` в нужной кассе.
3. Сверка кассы создаётся/обновляется.

### 1.4 Заказы 3D‑печати (калькулятор/проекты)
1. Клиент создаёт заказ в ЛК.
2. Заказ появляется в ЛК и в админке.
3. Статусы отображаются по‑русски и цветом.

### 1.5 “Оставить заявку”
1. Отправка заявки уходит в ТГ и Max.
2. В БД не создаются пользователь/чат/заказ.

## 2) Важные сценарии (P1)

### 2.1 Права доступа (RBAC)
1. Админ видит всё.
2. Менеджер: магазин/заказы/касса в рамках прав.
3. Склад: складские разделы, но без финансов.
4. Бухгалтер: касса/отчёты/логи.

### 2.2 Экспорт CSV
1. Экспорт заказов магазина.
2. Экспорт логов.

## 3) Нефункциональные проверки (P1/P2)
1. Безопасность: нет ключей/токенов в репозитории, секреты только в ENV.
2. Стабильность: lint/typecheck/unit tests/build проходят.
3. Производительность: основные страницы отвечают 200/3xx корректно.

## 4) Автоматизация
1. Unit tests: `npm run test`
2. Базовая сборка: `npm run lint && npm run typecheck && npm run build`
3. Smoke бизнес‑процессов (сервер/стенд): `node scripts/smoke-business-processes.js --cleanup`
