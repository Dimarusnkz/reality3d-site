# План оптимизации и модернизации системы

Документ фиксирует результаты аудита, приоритизацию улучшений и поэтапный план внедрения с метриками успеха.

## 1) Краткий срез текущего состояния

**Технологии**
- Next.js App Router (`app/`), TypeScript.
- Prisma: основной клиент + отдельные клиенты/схемы для SQLite/MySQL.
- Server Actions + Route Handlers (`app/api/*`).
- TailwindCSS v4.

**Критические риски (P0)**
- Утечка чувствительных полей пользователя в клиент из server component (передача `User` целиком в client component).
- Неполный CSRF для state-changing операций (часть server actions/handlers не проверяет CSRF).
- Upload API без строгой валидации типов/размеров и с захардкоженными путями.
- Multi-DB переключение через глобальную мутацию клиента (риск гонок/непредсказуемости).

**Baseline качества (на момент подготовки плана)**
- `lint/typecheck/tests/build` проходят.
- `npm audit fix` устранил уязвимости dev-зависимостей.

## 2) Цели и метрики успеха

**Безопасность**
- 100% state-changing операций защищены CSRF.
- 0 случаев утечки секретных/чувствительных полей в HTML/JSON (включая `password`, токены, служебные поля).
- Upload: allowlist расширений + лимиты по размеру, пути конфигурируются через env.

**Надёжность/масштабируемость**
- Rate-limiting работает корректно в multi-instance (подготовка к Redis/провайдеру).
- Multi-DB переключение не приводит к смешиванию запросов в рамках одного запроса/транзакции.

**Качество и поддерживаемость**
- `npm run lint` без предупреждений.
- `npm run typecheck` без ошибок.
- `npm run test:coverage` соответствует порогам проекта.
- `npm run build` проходит.

## 3) Приоритизация

### P0 (немедленно)
1) Устранить утечки пользовательских данных в клиент (DTO/select).
2) Расширить CSRF на все изменяющие операции (server actions + route handlers).
3) Усилить upload: валидация, конфигурация путей, защита CSRF/Origin.

### P1 (после P0)
1) Стабилизировать multi-DB переключение: убрать глобальную мутацию или зафиксировать клиента на запрос.
2) Distributed rate-limiting (Redis/провайдер) на auth/admin endpoints.
3) Наблюдаемость: структурные логи + расширение аудита.

### P2 (оптимизация)
1) Чат: уменьшить стоимость запросов, пагинация сообщений, отказ от агрессивного поллинга.
2) Индексы/оптимизация Prisma-запросов на «горячих» выборках.

## 4) Этапы и сроки (ориентир)

**Этап 1 (2–3 дня): Security P0**
- DTO/select для user, запрет сериализации секретных полей.
- CSRF-guard для всех state-changing действий.
- Upload hardening (валидация, env paths, CSRF/Origin).

**Этап 2 (3–5 дней): Multi-DB стабилизация**
- Рефакторинг доступа к Prisma: исключить глобальную мутацию «на лету» или зафиксировать клиента на запрос.
- Инварианты переключения (только admin, аудит, health-check, миграции).

**Этап 3 (5–7 дней): Abuse/bot protection + observability**
- Rate limiting (готовность к distributed), Turnstile на критичные точки.
- Расширение аудита и безопасные логи.

## 5) Definition of Done (для этапов)

- Реализованы изменения и добавлены тесты на ключевые гарантии.
- Линтер/типизация/тесты/сборка проходят.
- Документация обновлена: env-переменные, поведение CSRF, upload, DB режим.
