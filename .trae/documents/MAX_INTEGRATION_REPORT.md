# Отчёт проверки интеграции с MAX

Дата: 2026-03-17

## 1) Область проверки

Проверена текущая интеграция с MAX в части:
- хранения и управления получателями уведомлений (таблица `maxSubscriber`)
- отправки уведомлений в MAX (вызовы API `https://platform-api.max.ru`)
- административного UI для настройки/теста

Документация MAX API (ключевые требования, релевантные задаче):
- Для production рекомендован **Webhook** (long polling — только для dev/test).
- Лимит на `platform-api.max.ru` — **до 30 rps**.

## 2) Текущая реализация (как устроено сейчас)

**Точка отправки уведомлений**
- Уведомления отправляются из server action отправки заявки: `app/actions/requests.ts` вызывает `sendMaxMessage(...)`.

**MAX-клиент**
- Реализован в `lib/max.ts`.
- Аутентификация: заголовок `Authorization: <MAX_BOT_TOKEN>`.
- Отправка: `POST /messages`.

**Получатели**
- Хранятся в БД (Prisma model `maxSubscriber`).
- Админ-UI: добавление/удаление/тест отправки.

**Webhook / callbacks**
- В проекте отсутствует обработчик Webhook от MAX.

## 3) Проверка конфигурации подключения

### 3.1 Аутентификация / авторизация

**Используется**: `MAX_BOT_TOKEN`.

**Проблемы**
- P0: секреты передавались в client-компоненты админки (токен MAX и токен Telegram) и могли утечь в браузер.

**Статус**
- Исправлено: теперь в UI передаётся только признак “токен задан/не задан”, без токенов.

### 3.2 Webhook-и / callback URL

**Требование MAX**: для production рекомендуется Webhook.

**Проблемы**
- P2: Webhook не настроен/не реализован в проекте, поэтому нет корректного получения событий от MAX (если потребуется обработка входящих сообщений/кнопок).

**Рекомендации**
- Добавить `POST` endpoint вида `/api/max/webhook` (HTTPS only), валидировать подпись/секрет (если MAX предоставляет), логировать события, ограничить по IP/секрету.
- На платформе MAX настроить callback URL на этот endpoint.

### 3.3 Права доступа

- Server actions управления подписчиками и тестовой отправки доступны только ролям `admin/manager` и дополнительно защищены CSRF.

## 4) Тестирование отправки уведомлений

### 4.1 Формат/структура сообщений

**Проблема**
- P1: ранее в MAX отправлялся HTML (тот же шаблон, что для Telegram). Поддержка HTML в MAX не подтверждена.

**Статус**
- Исправлено: перед отправкой в MAX сообщение приводится к plain-text (HTML теги удаляются, `br/p` переводятся в перенос строки).

### 4.2 Обработка ошибок/исключений

**Что проверено**
- Ошибки сети/таймауты не должны “ронять” основной поток выполнения.
- Повтор при transient ошибках.

**Статус**
- Добавлены таймауты запросов.
- Добавлен ограниченный retry (1 повтор) для `429` и `5xx`, а также при исключениях.

### 4.3 Ограничение нагрузки

**Требование MAX**: до 30 rps.

**Статус**
- Реализована отправка пакетами с ограничением `MAX_SEND_RPS` (по умолчанию 25 rps) и паузой между пакетами.

## 5) Интеграционное тестирование и сценарии

Добавлены тесты уровня библиотеки отправки (`vitest`):
- успешная отправка
- retry по ошибке payload
- обработка таймаута/abort
- пакетирование по `MAX_SEND_RPS`

Файл: `lib/max.test.ts`.

Примечание: реальные end-to-end тесты с вызовом MAX API не запускаются в CI, так как требуют внешней сети и реальных токенов. Для e2e рекомендуется отдельный окружение/staging.

## 6) Выявленные проблемы и критичность

### P0
- Утечка секретов в клиент из админских страниц (MAX/Telegram токены) — устранено.

### P1
- Нет таймаута/ограничения rps при массовой рассылке — устранено.
- Формат сообщения (HTML) мог быть некорректен для MAX — устранено (plain-text).

### P2
- Отсутствует Webhook интеграция для production (по документации MAX рекомендован Webhook для стабильной работы).

## 7) Рекомендации

- Реализовать Webhook-приёмник (`/api/max/webhook`) и настроить его в MAX для production.
- Выделить единый слой “notifications” с типами событий (новая заявка/изменение статуса заказа/и т.п.) и шаблонами отдельно для Telegram и MAX.
- Добавить наблюдаемость: метрики/логи по результатам отправки (успешно/ошибка/код), без логирования секретов.

## 8) Подтверждение работоспособности критических функций

- Отправка уведомлений в MAX выполняется через `lib/max.ts` с таймаутом, retries и ограничением rps.
- Управление подписчиками и тестовая отправка защищены авторизацией (роль) и CSRF.
- Секреты не пробрасываются в client UI.
